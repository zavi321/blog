WordPress vs Custom Code: How to Choose for a Client Project

This question comes up on nearly every new project: build it in WordPress, or hand-code it from scratch? Both are legitimate choices, and freelancers who default to one or the other regardless of the project usually end up either overcharging for something simple or underdelivering on something that needed more flexibility. The right answer depends on the client, not on which one you personally prefer to build in.

What WordPress is actually good at

WordPress powers a large share of the web for a reason — it solves a specific problem extremely well: giving a non-technical person the ability to manage their own content after the developer is gone.

  • Client self-editing. A business owner can log in and update text, swap images, or add a blog post without calling a developer every time.
  • Plugin ecosystem. Contact forms, SEO tools, e-commerce (via WooCommerce), booking systems — most common features already exist as a plugin, cutting build time significantly.
  • Faster delivery for standard sites. A theme-based WordPress build for a typical small business site can often be delivered faster than the same site hand-coded from a blank file.
  • Lower cost for the client. Because less custom development time is needed, WordPress projects are usually priced lower, which matters for clients with a tight budget.

Where WordPress starts to work against you

  • Plugin bloat. Stacking too many plugins to get exactly the features a client wants often slows the site down and creates conflicts between plugins.
  • Security maintenance. WordPress sites need regular updates — core, theme, and plugins — or they become a common target for automated attacks. This is ongoing work, not a one-time build.
  • Design ceiling. Highly specific, unusual designs are possible in WordPress but often fight the platform, taking more time than building the same design from scratch.
  • Hosting requirement. WordPress needs a server running PHP and a database, which means it cannot use free static hosting like GitHub Pages.

What custom code is actually good at

  • Full control. Every line exists because it was written for this exact project — no unused plugin code, no theme bloat, nothing fighting the design.
  • Performance. A hand-coded static site is typically faster to load than an equivalent WordPress site, since there's no database query or plugin overhead on every page load.
  • Free hosting for static sites. Sites without a login system, e-commerce, or a database can be hosted for free on platforms like GitHub Pages, saving the client an ongoing hosting cost.
  • No plugin security surface. With fewer moving parts and no third-party plugin code, there's less that can go wrong or be exploited.

Where custom code starts to work against you

  • The client can't edit it themselves. Any text or image change means either giving the client basic HTML training or staying on retainer for edits — which is fine if priced for, frustrating if not discussed upfront.
  • Slower to build features from scratch. A booking calendar or a product catalog that a WordPress plugin handles in an afternoon might take days to hand-code and test properly.
  • Higher price point. More development hours generally means a higher quote, which can lose budget-conscious clients even when custom code is the better technical fit.

A simple way to decide

Instead of picking a default and building every project the same way, three questions usually settle it:

  1. Will the client update content themselves after launch? If yes, WordPress is usually the better fit unless the design is highly unusual.
  2. Does the site need e-commerce, bookings, memberships, or other complex features? If yes, WordPress plugins can save significant build time — unless the feature set is unusual enough that a plugin would need heavy customization anyway, in which case custom code may end up cleaner.
  3. Is the site purely informational — a brochure site, portfolio, or landing page — with no client-side editing needed? Custom code on free static hosting is often the leaner, faster, and cheaper choice here, for both the developer and the client.

What this means for pricing

These aren't just technical choices — they change the shape of the client conversation. A WordPress quote should account for ongoing maintenance (updates, backups, security) either as a line item or a clearly stated client responsibility. A custom-code quote should be clear about what happens when the client needs a text change six months later — whether that's included, billed hourly, or requires basic training up front.

Framing the choice this way, as a fit to the client's actual needs rather than a personal preference, tends to build more trust in a proposal than simply naming a tech stack and hoping the client understands why.

The takeaway

Neither WordPress nor custom code is universally "better" — they solve different problems. The freelancers who deliver the most value consistently are the ones who ask the right questions before quoting, rather than reaching for whichever stack they're most comfortable in by habit.
